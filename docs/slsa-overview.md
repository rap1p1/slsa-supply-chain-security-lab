# SLSA Framework Overview

## What is SLSA?

**SLSA** (Supply-chain Levels for Software Artifacts, pronounced "salsa") is a security framework created by Google and the Open Source Security Foundation (OpenSSF) that provides a checklist of standards and controls to prevent tampering, improve integrity, and secure packages and infrastructure.

## The Problem SLSA Solves

Traditional security focuses on finding vulnerabilities in **your code** (SAST/DAST). But modern attacks target the **supply chain**:

| Attack | What Happened | SLSA Prevention |
|--------|--------------|-----------------|
| **SolarWinds (2020)** | Attacker injected malware into the build system. Signed builds contained backdoor. | SLSA L3: Provenance would show the build was modified by unauthorized builder |
| **Codecov (2021)** | Attacker modified CI script to exfiltrate secrets | SLSA L3: Isolated builder prevents CI script tampering |
| **Log4Shell (2021)** | Critical vulnerability in ubiquitous library | SBOM: Instantly identify all affected deployments |
| **xz-utils (2024)** | Maintainer social engineering → backdoor in build process | SLSA L3: Provenance tracks exact build steps, detects anomalies |
| **npm packages (ongoing)** | Typosquatting, dependency confusion | SBOM + Policy: Verify all dependencies against known-good list |

## SLSA Levels Explained

```
Level 0: No guarantees
   ↓
Level 1: Documentation of the build process
   ↓  + Provenance exists (but may be self-attested)
Level 2: Tamper resistance of the build service
   ↓  + Provenance generated by hosted build service
   ↓  + Authenticated provenance
Level 3: Hardened builds ← THIS PROJECT
   ↓  + Provenance generated by ISOLATED builder
   ↓  + Non-falsifiable provenance
   ↓  + Builds run in ephemeral, isolated environments
Level 4: Two-party review (future, not yet defined)
```

### What Makes Level 3 Special?

| Requirement | Level 2 | Level 3 |
|------------|---------|---------|
| Provenance exists | ✅ | ✅ |
| Hosted build service | ✅ | ✅ |
| Authenticated provenance | ✅ | ✅ |
| **Isolated builder** | ❌ | ✅ |
| **Non-falsifiable provenance** | ❌ | ✅ |
| **Ephemeral build environment** | ❌ | ✅ |

The key difference: At Level 3, **even the repository owner cannot fake provenance**. The provenance is generated by a separate, trusted system (slsa-github-generator) that the developer cannot modify.

## How This Project Implements SLSA Level 3

### 1. Build Service (GitHub Actions)
- Hosted, authenticated build service 
- Ephemeral runners (each build gets a fresh VM) 
- Audit log of all builds 

### 2. Provenance Generation (slsa-github-generator)
- Runs as a **reusable workflow** in a separate job 
- Developer cannot modify the provenance generation code 
- Provenance signed with GitHub's OIDC identity 
- Records: source URI, commit hash, builder ID, build steps 

### 3. Signing (Cosign + Sigstore)
- Keyless signing via OIDC (no static secrets) 
- Signatures recorded in Rekor transparency log 
- Verifiable by anyone without special keys 

### 4. Verification (Kyverno)
- Admission webhook verifies before pod creation 
- Checks signature, provenance, and SBOM 
- Enforce mode: blocks non-compliant images

```json
  }

```

## Industry Adoption (2025)

- **US Executive Order 14028**: Requires SBOM for all software sold to the US government
- **EU Cyber Resilience Act**: Mandates supply chain security for all digital products in EU
- **PCI DSS 4.0**: Requires software composition analysis for payment systems
- **SOC 2 Type II**: Supply chain security is now a common audit requirement
- **NIST SP 800-218**: SSDF (Secure Software Development Framework) aligns with SLSA
